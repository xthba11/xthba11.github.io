# 个人博客完善规划

## 目标

把当前博客从“技术笔记集合”升级成一个更完整、更可信、更像真实工程师长期维护的个人技术博客。

核心方向不是堆更多文章，而是让内容更具体：有真实开发环境、真实板卡/芯片/工具链、真实问题、真实调试过程、真实取舍和复盘。读者看完后应该能判断：这个人真的做过、调过、踩过坑。

## 当前项目现状

### 已经做得不错的地方

- 技术方向清晰：嵌入式 MCU、RTOS、Linux 驱动、网络编程、eSSD 存储系统。
- 文章数量已有基础：`source/_posts` 下已有 22 篇文章，能支撑一个技术博客的初始规模。
- 主题配置比较完整：使用 Hexo + Butterfly，有分类、标签、目录、代码高亮、评论、站内统计、暗色模式等。
- 页面结构已经具备：首页、分类、标签、项目页、关于页、404 页面都有。
- 内容风格偏工程化：很多文章包含代码、流程图、常见坑、测试命令，不是纯概念介绍。

### 目前最像 AI/模板的地方

- 个人经历不够具体：关于页说“嵌入式工程师”，但缺少真实背景，例如做过什么板子、什么项目、调过什么外设、常用什么环境。
- 项目页还是“计划型”：大量内容是“计划文章”“后续补齐”，读者看不到已经完成的项目仓库、目录结构、运行截图、测试记录。
- 文章缺少现场感：多数文章是标准知识讲解，缺少“我当时为什么查这个问题”“在哪个环境复现”“最后怎么定位”的叙述。
- 图片资源单一：几乎所有文章都使用同一张 `embedded-lab-hero.png`，头像和 favicon 配置了路径但资源文件不存在。
- 部分内容标题过大：例如“从入门到实战”“详解”容易显得泛泛，建议换成更具体的工程标题。
- 代码注释不统一：文章中的关键代码有一些解释，但程序关键路径、错误处理、并发/时序边界还可以加更有价值的注释。

## 页面改造建议

### 关于页

当前关于页的优点是清楚，但还不够“有人味”。建议加入：

- 目前身份：学生/工程师/自学者/求职中等。
- 常用环境：Ubuntu 版本、开发板、MCU 型号、内核版本、工具链。
- 做过的真实项目：哪怕是课程设计、实验项目、个人 demo，也要写具体。
- 写博客的原因：为了复盘、求职展示、逼自己把知识讲清楚。
- 技术偏好背后的例子：例如“我遇到过一次 I2C ACK 问题，所以现在习惯先用逻辑分析仪确认波形”。

## 文章内容改造标准

### 标题标准

减少宽泛标题：

- 不推荐：`MCU开发专题：蓝牙开发从入门到实战`
- 推荐：`HC-05 透传模块调试记录：AT 模式、串口接线与常见乱码`

- 不推荐：`Linux网络开发专题：常见踩坑与解决方案`
- 推荐：`TCP 粘包问题复现：用 length-header 协议修掉半包解析`

- 不推荐：`Linux 字符设备驱动入门：从 open 到 ioctl`
- 推荐：`demo_chrdev 字符设备驱动：cdev 注册、ioctl 与用户态测试`

### 每篇文章至少补 3 类具体信息

- 具体环境：Ubuntu、GCC、Linux kernel、开发板、芯片型号、工具版本。
- 具体输入输出：命令、日志、抓包、dmesg、串口输出。
- 具体边界：失败条件、错误码、异常分支、并发/中断/超时情况。

### 增加“我踩过的坑”

比“常见坑”更像真人，因为它来自实际经历。

可以写成：

```markdown
## 我踩过的坑

### probe 一直不进

现象：

`dmesg` 里没有驱动日志，`ls /sys/bus/platform/drivers` 能看到驱动，但设备没有绑定。

定位：

最后发现设备树里的 `compatible` 写成了 `xthba,demo-gpios`，驱动里是 `xthba,demo-gpio`。

复盘：

以后写设备树和驱动匹配表时，先用 `grep compatible` 对一遍字符串。
```

## 程序关键部分注释规范

注释不要解释语法，而要解释意图、边界和风险。

### 不推荐的注释

```c
// 定义变量
int ret;

// 调用 read 读取数据
n = read(fd, buf, sizeof(buf));
```

### 推荐的注释

```c
/*
 * 非阻塞 fd 必须一直读到 EAGAIN。
 * 如果这里只读一次，在 ET 模式下剩余数据可能不会再次触发事件。
 */
while ((n = read(fd, buf, sizeof(buf))) > 0) {
    handle_frame(conn, buf, n);
}
```

### 驱动代码重点注释位置

- `probe`：资源获取顺序、失败回滚顺序。
- `remove`：为什么按这个顺序释放资源。
- `read/write`：用户态缓冲区、长度限制、阻塞/非阻塞行为。
- `ioctl`：命令号兼容性、参数结构体版本、用户态指针检查。
- 中断处理：中断上半部只做什么，下半部/工作队列做什么。
- 锁：保护哪个共享资源，是否可能在中断上下文使用。

### 网络代码重点注释位置

- `set_nonblock`：为什么 epoll 服务端必须配非阻塞。
- `accept` 循环：为什么要一直 accept 到 `EAGAIN`。
- `read` 循环：LT/ET 模式差异。
- `write`：短写、发送缓冲区、`EPOLLOUT` 注册时机。
- 连接对象：fd、状态、收发缓冲区、最后活跃时间。
- 协议解析：半包、粘包、非法长度、超时丢弃。

### RTOS/MCU 代码重点注释位置

- 中断回调：哪些操作不能在 ISR 里做。
- 任务优先级：为什么这个任务优先级更高。
- 队列长度：按什么频率估算。
- DMA buffer：缓存一致性、对齐、生命周期。
- 状态机：状态转换条件和异常恢复路径。
- 看门狗：每个任务如何证明自己还活着。

## 判断一篇文章是否“像真人写的”

写完后用这 8 个问题自查：

- 有没有具体硬件、系统或工具版本？
- 有没有一条能复制执行的命令？
- 有没有真实输出，例如日志、错误码、抓包或截图？
- 有没有解释为什么这样设计，而不是只说怎么写？
- 有没有失败记录？
- 有没有说明边界条件？
- 有没有把 AI 式空话换成自己的判断？
- 有没有指出哪些内容还没验证？

## 最小执行方案

如果只想先快速提升质感，按这个顺序做：

1. 改关于页，让读者知道你是谁、做过什么、常用什么环境。
2. 改项目页，把“计划”改成“当前状态 + 已验证内容 + 下一步”。
3. 选 2 篇代表文章补测试环境、日志、踩坑和注释。
4. 补头像、favicon、专题封面。
5. 修搜索路径、部署说明、缺失页面这些配置问题。

这 5 步做完，博客的可信度和个人感会明显提升。
